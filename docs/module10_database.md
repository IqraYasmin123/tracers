# Module 10 — Database Design

## Goal

A normalized PostgreSQL schema for users, cases, evidence, AI results, reports, and audit
logs — the persistence layer that Module 13 (Case Management) will wire into the FastAPI
backend built in Module 9.

## Design

```
users        — accounts and roles (admin / investigator / viewer)
cases        — forensic investigations, owned by a user
evidence     — uploaded images, belonging to a case
ai_results   — one TRACER analysis result per piece of evidence
reports      — generated PDF/DOCX reports, belonging to a case
audit_logs   — who did what, when (security/compliance trail)
```

See the ER diagram generated earlier in this conversation for the full relationship map.

## Normalization (3NF)

- Every foreign key references a primary key directly; no table duplicates data that
  belongs in another table (e.g. a case's title is never copied into its evidence rows).
- `ai_results` references `evidence`, not `cases` — an AI result belongs to one specific
  piece of evidence. Referencing `cases` directly would create a transitive dependency
  (case info reachable two different ways) and would break down the moment a case has
  multiple pieces of evidence with different results, which is the normal case.
- `ai_results.evidence_id` has a `unique=True` constraint — enforced at the schema level
  that one piece of evidence has at most one AI result, not just an application-level rule.
- JSON columns (`explanation_details`, `audit_logs.details`) hold genuinely variable-shape
  metadata for a single row, not multivalued attributes with independent identity — using
  JSON here is a deliberate modeling choice, not a normalization shortcut.

## Design Decisions

### SQLAlchemy models as the single source of truth

`database/models.py` is authoritative. `database/schemas/schema.sql` is **generated from**
the models (via `scripts/generate_schema_sql.py`), not hand-written — this guarantees the
two can never drift out of sync. Regenerate it any time `models.py` changes:
```bash
python scripts/generate_schema_sql.py
```

### Never hand-author Alembic migrations

`database/migrations/versions/..._initial_schema.py` was generated by running
`alembic revision --autogenerate`, then verified by actually applying it
(`alembic upgrade head`) against a real SQLite database and confirming all 6 tables and every
index were created correctly. For any future schema change: edit `models.py`, then run
`alembic revision --autogenerate -m "description"` again — never hand-write a migration.
Hand-transcribing ~40 columns across 6 tables invites subtle drift bugs that autogenerate
eliminates entirely.

### Testing without a real PostgreSQL server

`database/tests/test_models.py` runs against an in-memory SQLite database — fast (11 tests
in well under a second), no server setup needed for CI/development. This verifies the
schema's *logic* (relationships, unique constraints, cascade deletes) correctly, but SQLite
doesn't exercise every PostgreSQL-specific behavior (native `ENUM` storage, `JSONB`
indexing). Once you have a real PostgreSQL server running, it's worth spot-checking the
same test suite against it by setting `DATABASE_URL` to a real Postgres connection string —
the tests themselves don't need to change, only the database they run against.

### A path-resolution bug caught during development

Both `database/tests/test_models.py` and `scripts/generate_schema_sql.py` initially failed
with `ModuleNotFoundError: No module named 'database'` when run from certain directories —
Python only adds the *invoking* directory to its import path, not the repo root that
`database` actually lives under. Fixed with an explicit `sys.path` correction in both
`database/tests/conftest.py` and the script itself, verified to work regardless of which
directory they're invoked from. Same underlying lesson as Module 9's `python -m pytest`
fix — worth naming as a recurring category of bug across this project (Python import
resolution depends on *how* and *where* you invoke a script, not just what the code says).

## PostgreSQL Setup (Windows)

**Option A — Docker (recommended, avoids native install/PATH issues):**
```bash
docker run --name tracer-postgres -e POSTGRES_PASSWORD=changeme -e POSTGRES_DB=tracer_db -p 5432:5432 -d postgres:16
```
Then set in your `.env`:
```
DATABASE_URL=postgresql://postgres:changeme@localhost:5432/tracer_db
```

**Option B — Native installer:** download PostgreSQL from postgresql.org, run the installer,
remember the password you set for the `postgres` user, then use the same `DATABASE_URL`
format above with your actual password.

**Then run the migration:**
```bash
pip install -r requirements.txt
alembic upgrade head
```

## Testing Strategy

```bash
python -m pytest database/tests/ -v   # from repo root — 11 tests, <1s, SQLite in-memory
```
