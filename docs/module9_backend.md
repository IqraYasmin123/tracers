# Module 9 — FastAPI Backend

## Goal

Expose the AI engine (Modules 2-8) as a real, testable REST API — the bridge between the
React/Flutter clients and the AI pipeline. Authentication (Module 16) and persistence
(Module 10) are deliberately out of scope here; this module is the API surface and request
handling only.

## Design

```
app/main.py                        FastAPI app, middleware, exception handlers, routers
app/config.py                       Settings, loaded from .env
app/logging_config.py                Logging setup
app/dependencies.py                  get_pipeline_service() — the DI entry point
app/services/pipeline_service.py     TracerPipelineService — wraps the AI engine
app/schemas/inference.py             Pydantic request/response models
app/api/routes/health.py             GET /api/v1/health
app/api/routes/inference.py          POST /api/v1/analyze
```

## Endpoints (v0.1.0)

| Method | Path | Purpose |
|---|---|---|
| GET | `/` | Service info |
| GET | `/api/v1/health` | Health check |
| POST | `/api/v1/analyze` | Upload an image (+ optional caption) → detection + attribution + explanation |

Interactive docs are auto-generated by FastAPI at `/docs` once the server is running.

## Design Decisions

### Dependency injection for the AI pipeline — the core pattern of this module

`TracerPipelineService` is provided to routes via `Depends(get_pipeline_service)`, not
imported and called directly. This is the exact same principle behind `BaseVLM`/`BaseAttack`/
`BaseDetector` in the AI engine, applied at the API layer: tests substitute a
`FakePipelineService` (instant, in-memory, no model) via `app.dependency_overrides`, so the
**entire backend test suite (8 tests) runs in 0.07 seconds without importing torch at all.**
This is worth stating explicitly in your report as a deliberate, consistent architectural
choice carried through every layer of the project, not just the AI engine.

### Lazy loading, not eager loading, of the AI engine

`TracerPipelineService._ensure_loaded()` only loads CLIP and the trained detector on the
*first real call* to `analyze()`, not at app startup. This means:
- The FastAPI app itself starts instantly, even without a GPU or trained model present
- `pytest` never triggers a model load, since tests use the fake service
- A misconfigured deployment (missing detector weights) fails with one clear error on first
  real request, rather than crashing on startup with a less specific stack trace

### Why reconstruction has no endpoint yet

Diffusion reconstruction takes 30-60 seconds and needs a multi-GB model — fundamentally
unsuitable for a synchronous HTTP request in a real system. A production deployment would
run it as a background job (e.g. via Celery + a task queue, with the client polling a
`/jobs/{id}` endpoint or receiving a webhook) rather than blocking a request thread for a
minute. Implementing that job-queue infrastructure is out of scope for the current module
list, so reconstruction stays a Colab-only capability (Module 7's notebook) for now — this is
an honest architectural limitation worth naming directly in your report, not something to
gloss over.

### Error mapping: RuntimeError → 503, ValueError → 422

Every AI-engine exception (`VLMLoadError`, `DetectionError`, `AttributionError`, etc.)
subclasses `RuntimeError` — these represent the AI engine not being ready or configured
correctly, mapped to HTTP 503 ("service temporarily unavailable"). `ValueError` (e.g. from a
config dataclass's `__post_init__`) represents bad configuration/input, mapped to 422. This
means every custom exception defined across Modules 2-8 gets sensible HTTP semantics for
free, without the backend needing to know about each one individually.

### A documented piece of technical debt: the `src` package name

`TracerPipelineService` imports from the AI engine's `src` package (literally named `src`,
from `ai-engine/src/`) — not a more descriptive name like `tracer_ai`. This was carried over
from Modules 2-8's development; renaming it now would touch ~40 files for a purely cosmetic
improvement, with real risk of breaking working code. Documented here explicitly so it reads
as a conscious trade-off, not an oversight, if it comes up in your viva.

## Local Setup (Windows)

```bash
cd backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt

# For real inference (not needed just to run tests):
pip install -r ../ai-engine/requirements.txt
```

**To enable real detection**, download `entropy_detector.joblib` (and optionally
`attack_classifier.joblib`) from your Google Drive (`MyDrive/TRACER/models/`, saved by
Module 5's notebook) into `ai-engine/models/` locally.

**Run the server:**
```bash
uvicorn app.main:app --reload
```
Visit `http://127.0.0.1:8000/docs` for interactive Swagger UI — you can upload an image and
test `/api/v1/analyze` directly from the browser, no separate HTTP client needed.

**Performance note:** without a GPU, CLIP inference on your local Windows machine runs on
CPU — workable for detection/attribution (a few seconds per request), but reconstruction
(not exposed as an endpoint anyway, see above) would be impractically slow on CPU.

## Testing Strategy
